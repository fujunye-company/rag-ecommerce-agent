   228|## 设计原则
   229|
   230|1. **MVP向下兼容全量**：所有全量模块的文件路径和接口名在MVP阶段即预留，确保后期无需重构目录。
   231|2. **分层严格**：api/ 不写业务 → services/ 承担逻辑 → schemas/ 校验输出。
   232|3. **文件粒度 ≤50行原则**：每个文件单一职责，方便 CC 单文件修改（~60s）。
   233|4. **命名与规约一致**：遵循开发规约 v2.0 §3.4 和 §4.2 命名规范。
   234|5. **测试与源文件对应**：tests/ 下文件名 = test_ + 源文件名。
   235|6. **数据与代码分离**：种子数据和测试用例放在 data/，不散落在代码中。
