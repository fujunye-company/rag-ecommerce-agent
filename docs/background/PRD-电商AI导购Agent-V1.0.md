## 1. 业务背景

### 1.1 行业大势：AI 正在重写电商入口

2024–2026 年，电商行业最重要的范式变化是：**用户找商品的入口，正在从「搜索框」迁移到「对话框」**。这并非孤立的产品迭代，而是巨头同步发生的战略转向：

- **海外**：Amazon 于 2024 年 2 月发布 Rufus，2026 年 5 月将其整合为「Alexa for Shopping」；OpenAI 于 2025 年 4 月在 ChatGPT 上线购物推荐，9 月推出「Instant Checkout」实现 ChatGPT 内端到端下单（Etsy / Shopify）。
- **国内**：字节豆包于 2025 年 10 月接入抖音商城商品推荐，11 月上线商品卡，2026 年 3 月放量内测豆包 App 内直接下单、支付（无需跳转抖音）；阿里通义千问于 2026 年 1 月全面接入淘宝闪购、支付宝、飞猪、高德等全生态业务；京东于 2025 年 12 月底开启「京东 AI 购」独立 App 内测。

这些动作背后反映出行业共识：随着大模型能力成熟与对话式交互被大众接受，**AI 导购正在成为继搜索框、信息流、短视频之后的下一代电商核心入口**。流量分配规则、商品曝光逻辑、用户决策路径都在被重构，电商正经历从"信息搜索"到"辅助决策"的代际跨越。

### 1.2 市场规模

根据 Fortune Business Insights 发布的《Artificial Intelligence (AI) in Retail Market》报告：

| 指标 | 数据 |
|------|------|
| 2025 年全球 AI 零售市场估值 | 约 120–130 亿美元 |
| 2026 年规模 | 165.4 亿美元 |
| 2034 年预测 | 1058.8 亿美元 |
| 复合年增长率（2026–2034） | **CAGR 26.10%** |
| 增长最快地区 | 印度 |

报告中"个性化推荐与客服自动化"是增长最快的细分赛道，AI 导购恰是其核心载体。

### 1.3 技术演进：RAG 进入 Agentic 阶段

- **2020 年**：Lewis 等人提出 RAG（检索增强生成），通过"检索 + 生成"组合解决大模型的幻觉与时效性问题。
- **2023–2024 年**：行业广泛采用 Advanced RAG，引入查询重写、重排序、过滤等步骤。
- **2025 年起**：RAG 演进为 **Agentic RAG**——由 AI Agent 自主决定何时检索、检索什么、如何拼装上下文、是否需要多轮检索。LlamaIndex 等框架原生支持该模式。

本项目所采用的 **LangGraph + LlamaIndex + Qdrant** 技术组合，正是 Agentic RAG 在电商场景下的典型工程化形态。

### 1.4 一句话总结

> 在 AI 重写电商入口的窗口期，本项目以 Agentic RAG 为核心，构建一个**深度理解商品、理解用户、辅助决策**的电商 AI 导购 Agent，为电商从"信息搜索"向"辅助决策"的代际跨越提供可复用的工程实践参考。

## 2. 用户痛点

### 2.1 消费者痛点：决策疲劳已成结构性问题

**痛点 1：信息过载，决策疲劳**

- 典型电商平台 SKU 数百万计，用户平均浏览 20–50 个商品后放弃率显著上升。
- "决策疲劳"（Decision Fatigue）已被学术界证实：选项过多会导致用户决策质量下降、放弃选择或做出次优选择。
- 商品详情页堆砌参数、评价、促销规则，用户难以高效抓取关键信息。

**痛点 2：跨商品对比效率低**

- 用户为了在 A、B 两款商品之间做选择，往往需要切换多个详情页、对比多个参数表，耗费 5–15 分钟仍可能犹豫不决。
- 用户的真实问题往往是："这两个差别大吗？"、"我这种情况适合哪个？"——而非简单的参数列表。

**痛点 3：搜索词难以表达真实需求**

- 用户的需求往往是模糊的、场景化的："送闺蜜的圣诞礼物"、"适合北方冬天通勤穿的鞋"、"宝宝湿疹用的面霜"。
- 传统搜索引擎需要用户把模糊需求翻译成关键词，且只能命中商品标题或属性，理解不了场景。

**痛点 4：客服响应慢、回答浅**

- 人工客服响应慢、不在线、回答模板化；传统 chatbot 只能处理预设 FAQ，遇到"这款和那款比哪个更适合我"就失效。
- 部分品类（家电、母婴、3C）用户决策周期长，需要的不是一次性回答，而是**多轮、有上下文记忆的咨询**。

**痛点 5：决策信息不充分导致退货**

- 服装类目退货率约 30%，3C 类目约 10%。其中相当部分源于购买前对商品的理解偏差（尺寸、质感、功能符合度），本质上是决策环节信息不足。

### 2.2 商家侧痛点

- **客服人力成本高**：高峰期客服排队严重，但 70%+ 的问题是重复性的商品咨询。
- **商品教育成本高**：商家精心制作的商品详情页、评测内容、营销文档，用户往往不读，只看标题图和价格。
- **个性化推荐难做**：现有平台推荐多基于行为，难以基于"实时对话中的需求"做精准匹配。

### 2.3 现有 AI 导购的不足

即便是头部产品，仍存在明显改进空间：

| 不足 | 表现 |
|------|------|
| 知识库封闭 | 多数 AI 导购只能用平台自有商品数据，不支持商家上传非结构化营销文档 |
| 多模态支持弱 | 国内大部分产品仍以文字为主，对"拍照问商品"的支持不深入 |
| 缺乏端到端评测 | 产品上线后缺少持续的质量评测与反馈闭环机制 |
| 推荐缺乏可解释性 | 推荐结果常常是"黑箱"，用户不知道为什么是这几款 |
| 多轮一致性差 | 第三轮对话与第一轮可能给出矛盾的推荐 |

## 3. 市场竞品分析

### 3.1 海外标杆

#### Amazon Rufus（→ Alexa for Shopping）

- **发布时间**：2024 年 2 月 1 日（Beta），2026 年 5 月 13 日更名为 Alexa for Shopping
- **架构**：明确的 RAG 架构，训练数据 = Amazon 商品目录 + 用户评价 + 社区 Q&A + 网络公开信息
- **四大核心能力**：
  1. **品类导购研究**："买耳机要考虑什么？"
  2. **商品对比**："越野跑鞋和公路跑鞋有什么区别？"
  3. **具体商品咨询**："这款耐用吗？"
  4. **个性化推荐**：基于对话上下文动态推荐
- **可借鉴点**：
  - 不做独立 App，深度嵌入现有购物流程
  - 三层训练数据：结构化（商品目录）+ 半结构化（评价/问答）+ 非结构化（网络信息）
  - 从泛到精的对话设计：品类研究 → 商品对比 → 决策辅助
  - 分批灰度发布策略

## 4. 产品定位

### 4.1 一句话定位

> 本产品是一个面向移动端用户的、可由商家定制知识库的、具备多模态多轮对话能力的电商 AI 导购 Agent。

### 4.2 产品愿景

让消费者"像问朋友一样问商品"，让商家"上传文档就能拥有专属 AI 导购"。

### 4.3 Agent 能力分级

参考行业通用 Agent 能力分级框架，本项目定位为 **Level 2–3**：

| 等级 | 能力描述 | 本项目是否覆盖 |
|------|----------|----------------|
| L1 | 问答助手（FAQ Bot） | ✅ 基础能力 |
| L2 | 导购顾问（理解需求 + 多轮 + 推荐） | ✅ 核心能力 |
| L3 | 决策代理（对比 + 给出购买建议） | ✅ 核心能力 |
| L4 | 自主交易（自主比价 + 下单） | ❌ MVP 不涉及 |

### 4.4 目标用户

#### 4.4.1 C 端核心用户（消费者）

| 用户类型 | 占比（估） | 核心需求 | 典型场景 |
|----------|-----------|----------|----------|
| 效率型 | 40% | 快速找到最佳商品 | "推荐 2000 元以内的降噪耳机" |
| 探索型 | 35% | 不确定选什么，需要引导 | "送闺蜜的圣诞礼物" |
| 比价型 | 15% | 对比性价比 | "A 和 B 哪个更值" |
| 社交种草型 | 10% | 受内容种草，转向 AI 求证 | "小红书说这款好用是真的吗" |

#### 4.4.2 B 端用户（商家）

- **中腰部商家**：缺乏自建 AI 客服能力，希望低成本接入
- **品牌商家**：希望让 AI 准确传达品牌定位与产品卖点
- 商家的核心诉求是**上传文档即可拥有专属导购**，不需要懂 AI 工程

### 4.5 核心价值主张

| 用户 | 价值 |
|------|------|
| 消费者 | 减少决策时间 50%+，决策更自信，退货率降低 |
| 商家 | 客服成本下降 30%+，商品教育效率提升，转化率提升 |
| 平台 | 提升用户停留时长与 GMV，构建对话式商务新入口 |

### 4.6 差异化策略

| 维度 | 头部竞品 | 本产品（暂定） |
|------|----------|----------|
| 知识来源 | 封闭，仅平台数据 | **开放，商家可上传非结构化文档** |
| 评测 | 未公开 | **内建 RAGAS 评测 + 反馈闭环** |
| 可解释性 | 弱，黑箱推荐 | **每条推荐附带来源引用** |
| 多模态 | 文字为主 | **文字 + 图片双模态原生支持** |
| 工程参考 | 闭源 | **作为工程参考实现** |

## 5. 核心使用场景

以下 4 个场景按使用频率排序，是 MVP 必须覆盖的核心场景。

### 场景 1：品类研究（"我想买 X，但不知道选什么"）

> **用户故事**：小王想买一款适合通勤用的降噪耳机，预算 1500 元，但市面上型号太多。她打开本产品（暂定），发消息："想买降噪耳机，1500 内，主要通勤用。"

**期望交互**：
1. Agent 先确认 1–2 个关键变量（"主要在地铁还是户外用？""有线还是无线？"）
2. 给出 3–5 款候选商品卡片，附"为什么推荐"
3. 用户可继续追问任一款

**关键能力**：意图理解、多轮澄清、个性化推荐、商品卡片渲染

### 场景 2：商品对比（"A 和 B 哪个更适合我？"）

> **用户故事**：小李在两款扫地机器人之间犹豫，让 Agent 帮忙对比。

**期望交互**：
1. 用户输入两款商品名称或链接
2. Agent 返回**对比卡片**：分维度（吸力、续航、拖地、价格、保修）并标出差异
3. Agent 给出"哪种用户更适合 A、哪种适合 B"的判断

**关键能力**：多文档检索、结构化对比、可解释推荐

### 场景 3：图片提问（多模态）

> **用户故事**：小张刷到一双别人穿的鞋，截图发给 Agent："找一双类似的，预算 500 内。"

**期望交互**：
1. 用户上传图片 + 文字
2. Agent 识别图片中的商品类型、款式特征
3. 返回 3–5 款相似商品卡片

**关键能力**：图片理解（多模态 LLM）、以图搜商品、跨模态检索

### 场景 4：场景化购物（"送 X 什么礼物"）

> **用户故事**：小陈想给妈妈 60 岁生日送礼物，预算 500–1500，希望实用。

**期望交互**：
1. Agent 主动追问妈妈的喜好/健康状况
2. 给出 3 个维度的推荐：实用型、健康型、情感型
3. 每个维度 2 款商品

**关键能力**：场景理解、需求扩展、分类推荐

## 6. MVP 范围

### 6.1 范围原则

比赛周期内（约 2–4 周），MVP 必须满足：
- **可演示**：在一台手机上完整跑通核心场景
- **可评测**：能给出量化的质量指标（准确率、检索精度等）
- **可扩展**：架构可平滑演进到生产环境

### 6.2 知识库素材来源

赛题明确要求"上传非结构化的商品详情与营销文档构建专属知识库"，其语义边界是 **商家侧用于描述商品、解释卖点、支持营销转化的非结构化资料**。MVP 阶段采用以下两条来源路径：

#### 优先方案：Hugging Face 公开数据集

项目组采用 **[`superlinked/external-benchmarking`](https://huggingface.co/datasets/superlinked/external-benchmarking)** 数据集的 `benchmark-100k` 子集（10 万条样本，约 2.3 GB）作为知识库主力素材源。

**数据溯源**：源自 Amazon Reviews 2023（UCSD McAuley Lab），由 Superlinked 重新构建用于向量检索 benchmark。许可证 Apache 2.0，可用于比赛及研究用途。

**字段构成**：

| 字段 | 类型 | 在本项目中的用途 |
|------|------|------------------|
| `parent_asin` | string | 商品唯一 ID |
| `title` | string | 商品标题 |
| `description` | string（最长 90k 字符） | **核心非结构化商品详情字段，作为知识库主体入 Qdrant** |
| `main_category` | string（约 40 个一级类目） | 类目过滤维度 |
| `categories` | string | 详细类目路径 |
| `price` | float | 价格（入 PostgreSQL 商品表） |
| `average_rating` / `rating_number` | float | 评分统计（仅数值，无评价正文） |
| `image_url` | string | 商品主图链接，支持多模态检索 |
| `value` | List[Float64]，4154 维 | 预计算融合向量（1 类目 + 3 数值 + 3 文本 + 1 图像） |

**类目覆盖**（未做筛选）：Books、Automotive、Tools and Home Improvement、All Beauty、Electronics、Software、Health and Household 等。

**关键工程优势**：
- ✅ 内置预计算向量（4154 维，含文本 + 图像 + 数值 + 类目融合），可直接灌入 Qdrant，省去自行 Embedding 的开销（如需保留灵活性，也可仅用 `description` 字段重新嵌入）
- ✅ 同时含文本（`description`）与图片（`image_url`），原生支持多模态检索
- ✅ 数据集**仅含评分统计、不含用户评价正文**，与"商家侧非结构化文档"的边界严格对齐，无需额外过滤

**已知限制 → 应对**：
- 原始数据为英文 → MVP 演示阶段可对用户输入做翻译预处理；或保留原文检索 + 让 LLM 在生成阶段产出中文回答
- 严格意义的"营销文案"字段缺失（`description` 兼具商品介绍与卖点说明，但缺品牌故事、场景化营销内容）→ 由下方兜底方案的 LLM 生成补足

#### 兜底方案：LLM 生成的模拟商品详情与营销文档

在真实素材覆盖不足、字段缺失或样本数量不够时，由 LLM 基于真实商品的核心参数生成：
- 商品详情文档（800–1500 字 / 篇）
- 营销文案 / 卖点说明（500–1000 字 / 篇）
- 品牌故事 / 使用场景描述

生成内容元数据中标注 `synthetic: true`，便于评测时区分。

#### 明确不采用

- ❌ **用户评价数据**（属于平台侧 / 用户侧数据，不属于商家专属知识库范畴，与赛题"商家上传"语义不符）
- ❌ 爬取受反爬协议保护的电商平台内容
- ❌ HTML 网页直接抓取（格式不可控，MVP 阶段不予支持）

## 7. 详细功能需求

### F1 — 多模态对话输入

**功能描述**：用户可在对话框输入文字、上传图片，或文字 + 图片组合输入。

**需求细节**：
- 输入框支持纯文本，长度上限 500 字
- 支持上传图片：JPG/PNG/WEBP，单张 ≤ 5MB，单次最多 3 张
- 上传中显示进度，失败可重传
- 客户端做基础图片压缩（最长边 1024px）后再上传，减少网络耗时

**异常处理**：
- 网络异常：本地缓存输入内容，提示重试
- 图片格式不支持：友好提示
- 内容触发安全过滤：返回"换个问题试试"

**验收标准**：
- 用户从点击发送到首字出现 ≤ 3 秒（P95）
- 图片上传成功率 ≥ 99%

### F2 — 流式输出

**功能描述**：基于 SSE，AI 回复逐字/逐句渲染。

**需求细节**：
- 后端使用 SSE（FastAPI 的 `EventSourceResponse`）推送
- 客户端边接收边渲染，支持中途打断（用户点"停止生成"）
- 流式过程中显示"AI 正在思考..."→"AI 正在生成..."状态

**性能要求**：
- 首字时间（TTFT）≤ 3s（P95）
- 流式速率 ≥ 20 tokens/s

### F3 — 多轮对话与上下文记忆

**功能描述**：Agent 能记住会话内的上下文，支持多轮追问。

**需求细节**：
- 会话级记忆：单次会话内 Agent 知道用户此前的所有发言与自己的回复
- 上下文窗口：保留最近 N 轮（N 视模型而定，默认 10 轮）
- 跨会话不记忆（MVP 不做长期记忆，避免隐私合规复杂度）
- 会话标题自动生成（取前 3 句的概括）

**LangGraph 工作流**（核心节点）：
```
[入口] 接收用户输入
   ↓
[节点 1] 意图理解（识别：品类研究 / 商品对比 / 场景化购物 / 多模态识别）
   ↓
[节点 2] 信息完整性检查（是否需要追问）
   ├──是→ 生成追问 → [出口]
   └──否→ ↓
[节点 3] 调用 RAG 工具（LlamaIndex）
   ↓
[节点 4] 调用结构化数据工具（PostgreSQL 商品库）
   ↓
[节点 5] 整理结果 → 调用 LLM 生成回复
   ↓
[节点 6] 内容安全过滤
   ↓
[出口] 流式输出（含商品卡片 JSON）
```

### F4 — 商品卡片实时渲染

**功能描述**：AI 推荐商品时，输出包含商品卡片的结构化结果，客户端渲染为可交互卡片。

**卡片字段**：
- 商品图（缩略图 + 主图）
- 名称、品牌、价格、原价（如有促销）
- 关键参数（3–5 条）
- **推荐理由**（来自 LLM 生成）
- **来源引用**（哪条评价 / 哪份文档 / 哪个详情页）
- 操作按钮：详情、加入对比、收藏

**渲染规则**：
- 卡片由后端以 JSON 协议返回，客户端解析渲染
- 单次回复最多 5 张卡片（避免视觉疲劳）
- 卡片渲染与文字流式同步（不阻塞文字输出）

### F5 — 商品对比卡片

**功能描述**：当用户触发对比意图时，返回结构化对比表。

**展示形式**：
- 横向 2–3 列对比
- 行：价格、关键参数（6–8 项）、卖点摘要（来自商家营销文档）
- 差异自动高亮（用颜色区分谁更优）
- 底部附 Agent 的"适用人群判断"

### F6 — 商家知识库上传

**功能描述**：商家可在 B 端上传**商品详情、商品说明、营销文案、宣传资料、图文资料等非结构化文档**，构建专属知识库。上传内容应聚焦于"描述商品、解释卖点、支持营销转化"的商家侧资料。

**支持上传格式**：
- PDF
- Word（DOCX）
- Markdown（MD）
- 纯文本（TXT）
- 图片（JPG / PNG，自动 OCR 提取文字）
- **图文混排资料**：PDF / Word / Markdown 中嵌入商品图、参数图、卖点图等

**明确不支持（MVP 阶段）**：
- ❌ HTML 网页 / URL 链接抓取（格式不可控，不予支持）
- ❌ 用户评价 / 平台评论数据导入（不属于商家专属知识库范畴）
- ❌ CSV / JSON 等结构化参数表（应通过商品库导入流程，而非知识库通道）

**处理流程**：
1. 上传 → 临时存储
2. 文档解析（PDF/DOCX/MD/TXT 抽取文本，图片走 OCR，图文混排同时抽取文本与图描述）
3. 分块（按语义段落 / 商品粒度切分，而非固定长度）
4. Embedding（使用 text-embedding 模型）
5. 入库（向量入 Qdrant，元数据入 PostgreSQL）
6. 状态回写商家端（"已生效""失败"）

**质量保障**：
- 上传后系统自动生成 5 条 sample 问题并测试回答，提示商家确认效果
- 商家可标记"该回答不准"，触发知识库迭代

### F7 — RAG 检索 + 引用来源标注

**功能描述**：所有商品咨询类回答必须基于检索结果，并附原文引用。

**检索流程**（LlamaIndex 实现）：
1. **查询重写**：LLM 将口语问题改写为检索友好的查询
2. **混合检索**：向量检索（Qdrant 语义） + 关键词检索（PostgreSQL 全文）
3. **重排序**：基于相关性 + 时效性 + 商家权重
4. **过滤**：基于价格区间、品类、品牌等元数据
5. **取 Top-K**：默认 K=5 喂给 LLM

**引用规范**：
- 回答中带 `[1]`、`[2]` 标号
- 卡片底部展示引用列表（来源类型 + 商品名 + 文档名）

### F8 — 用户反馈

**功能描述**：用户可对 AI 回复给予显式反馈。

**显式反馈**：
- 点赞 / 点踩按钮
- 点踩需选择原因：信息不准、不相关、太长 / 太短、其他
- 可附文字反馈

**隐式反馈**：
- 卡片点击率
- 对话完成率（是否中途退出）
- 后续是否进入商品详情页

**反馈数据用途**：
- 离线评测样本积累
- Prompt 与 RAG 检索策略优化
- 知识库迭代依据

### F9 — 离线质量评测（基于 RAGAS）

**评测维度**（参考 RAGAS 框架）：

| 维度 | 定义 | MVP 目标 |
|------|------|----------|
| Faithfulness（忠实度） | 回答是否忠于检索到的上下文（不幻觉） | ≥ 0.85 |
| Answer Relevancy（相关性） | 回答是否切题 | ≥ 0.85 |
| Context Precision（精准度） | 检索结果的相关性排序 | ≥ 0.80 |
| Context Recall（召回率） | 是否检索到所有相关文档 | ≥ 0.75 |

**电商特有补充指标**：
- 推荐准确率 P@5（前 5 个推荐中真正相关的比例）≥ 0.70
- 多轮一致性（前后推荐不矛盾）≥ 0.90
- 商品卡片字段完整率 ≥ 0.95

**评测数据集**：
- 自建 200 条标准测试问答对（覆盖 4 个核心场景）
- 每条标注：标准答案 + 应检索文档 + 期望商品卡片
- 测试集每轮迭代复用，对比迭代前后效果

### F10 — 内容安全过滤

**过滤维度**（按《生成式人工智能服务管理暂行办法》第四条）：
- 政治敏感、违法违规
- 暴力、色情、虚假信息
- 歧视性表述（民族 / 信仰 / 性别 / 地域 / 年龄 / 健康）
- 引诱性消费、虚假宣传

**实现方式**：
- 输入端：用户输入先过敏感词词典 + LLM 分类器
- 输出端：LLM 生成后再过一次安全过滤
- 失败处理：返回"换个问题试试"，记录日志

### F11 — AI 身份标识 + 投诉举报

**透明度要求**（《暂行办法》第四条 + 第十五条）：
- 首页 / 对话头部明确"我是 AI 导购助手"
- 每条 AI 回复尾部小字"由 AI 生成，仅供参考"
- "设置" 中含投诉举报入口，公布处理流程与时限（≤ 5 个工作日）

> **⚠️ MVP 范围说明**：本功能在 MVP 阶段**仅做 UI 文字展示**，不接入实际工单系统、客服系统或邮件后台。点击"提交"后只做本地日志记录与"已提交，我们将在 5 个工作日内处理"的提示，便于演示合规姿态。真实投诉处理流程留待产品正式商业化阶段对接。

### 9.3 LangGraph 工作流节点设计

| 节点 | 输入 | 输出 | 实现 |
|------|------|------|------|

| `retrieve_kb` | 重写后的查询 | Top-K 文档 | LlamaIndex |
| `query_db` | 结构化条件 | 商品列表 | PG SQL |
| `compose_answer` | 检索结果 + 上下文 | LLM 回复 + 卡片 JSON | LLM |
| `safety_check` | LLM 输出 | 通过 / 拒绝 | 安全分类器 |
| `stream_output` | 安全通过的回复 | SSE 流 | FastAPI |

### 9.4 数据流（以"推荐降噪耳机"为例）

```

  ↓

  ↓
need_clarify → 「信息够了」
  ↓
retrieve_kb（"地铁通勤降噪耳机 1500 内"）→ 5 篇商品文档
query_db（filter: 类目=耳机 AND 价格<=1500 AND 类型=降噪）→ 12 款 SKU
  ↓
compose_answer → 选 3 款 + 推荐理由 + 商品卡片 JSON
  ↓
safety_check → 通过
  ↓
stream_output → SSE 推送 → App 渲染
```

## 10. 质量评测与反馈闭环

### 10.1 评测体系总览

本项目构建**三层评测体系**，自下而上递进：

| 层级 | 评测对象 | 频率 | 数据来源 |
|------|----------|------|----------|
| 第 1 层：离线评测 | 单次回答的质量 | 每次迭代 | 标准评测集（200 条 QA 对） |
| 第 2 层：在线评测 | 不同策略的相对优劣 | 演示期 / 试运行期 | A/B 分流真实流量 |
| 第 3 层：用户反馈 | 真实使用质量 | 持续 | 显式 + 隐式行为数据 |

评测指标分为四个大类：**RAG 核心指标、电商场景指标、性能体验指标、安全合规指标**。

---

### 10.2 RAG 核心评测指标（基于 RAGAS 框架）

RAGAS（Retrieval-Augmented Generation Assessment）是社区事实标准的 RAG 评测框架，本项目采用其四大核心指标。各指标取值范围均为 **[0, 1]，越大越好**。

| 指标 | 定义 | 计算方式 | 数据来源 | MVP 目标 |
|------|------|----------|----------|----------|
| **Faithfulness（忠实度）** | 回答中的每个事实性声明是否都能从检索到的上下文中得到支持，衡量"幻觉"程度 | (1) 用 LLM 将回答拆解为 n 个原子声明 c₁…cₙ；(2) 对每个 cᵢ 判断是否能从上下文推导出，得二元标签 supported(cᵢ) ∈ {0,1}；(3) Faithfulness = 被支持声明数 ÷ n | 检索上下文 + 生成回答 | ≥ 0.85 |
| **Answer Relevancy（回答相关性）** | 生成的回答与用户原始问题的语义相关程度，惩罚"答非所问"与冗余信息 | (1) 用 LLM 基于回答反向生成 N 个可能问题 q'₁…q'ₙ（N 默认=3）；(2) 计算每个 q'ᵢ 与原始问题 q 的 embedding 余弦相似度；(3) AR = (1/N) × Σ cos(emb(q), emb(q'ᵢ)) | 原始问题 + 生成回答 | ≥ 0.85 |
| **Context Precision（上下文精准度）** | 检索返回的 K 个文档中，真正相关的是否排在前面（rank-aware） | (1) 对 Top-K 标注相关性 vₖ ∈ {0,1}；(2) 各位置 Precision@k =（前 k 中相关数）÷ k；(3) CP@K = Σ(Precision@k × vₖ) ÷ Σ vₖ | 检索结果 + 人工/LLM 标注 | ≥ 0.80 |
| **Context Recall（上下文召回率）** | 标准答案中所需的信息是否都被检索到，衡量是否"漏召" | (1) 将标准答案拆解为 m 条事实陈述 g₁…gₘ；(2) 对每条 gⱼ 判断是否能在上下文中找到支持；(3) CR = 被支持事实数 ÷ m | 标准答案 + 检索结果 | ≥ 0.75 |

> **算例（Faithfulness）**：回答"索尼 WH-1000XM5 续航 30 小时、支持 LDAC"被拆解为 2 个声明，两条均能在上下文中找到 → Faithfulness = 2/2 = **1.0**。
> **算例（Context Precision）**：K=5，相关性序列 [1, 1, 0, 1, 0] → P@1=1, P@2=1, P@4=0.75 → CP@5 = (1+1+0.75)/3 ≈ **0.92**。

### 10.3 电商场景特有评测指标

| 指标 | 定义 | 计算方式 | MVP 目标 |
|------|------|----------|----------|
| **意图分类准确率（IA）** | LangGraph 的 `intent_classify` 节点输出与标注意图一致的比例。意图共 4 类：品类研究 / 商品对比 / 场景化购物 / 多模态识别 | IA = 正确分类样本数 ÷ 评测集总样本数 | ≥ 0.90 |
| **Precision@K**（推荐精确率） | 推荐列表前 K 个商品中相关的占比 | P@K =（Top-K 中相关数）÷ K。K 取 3（首屏）与 5（完整列表） | P@3 ≥ 0.75，P@5 ≥ 0.70 |
| **Recall@K**（推荐召回率） | 推荐列表前 K 个中命中的相关商品占应推荐总数的比例 | R@K =（Top-K 中相关数）÷ 应推荐相关商品总数 | R@5 ≥ 0.60 |
| **MRR（平均倒数排名）** | 第一个相关商品出现位置的倒数的平均，衡量用户能在第几位看到正确推荐 | MRR = (1/\|Q\|) × Σᵢ (1 ÷ rankᵢ)，rankᵢ 为第 i 条 query 中首个相关项的位置 | ≥ 0.70 |
| **NDCG@K**（归一化折损累积增益） | 考虑排名位置加权的累积相关度，越靠前的相关结果贡献越大；rel 取 0/1/2 | DCG@K = Σᵢ(relᵢ ÷ log₂(i+1))；NDCG@K = DCG@K ÷ IDCG@K | NDCG@5 ≥ 0.80 |
| **多轮一致性（MTC）** | 多轮对话中前后给出的推荐 / 判断是否一致（无矛盾）。典型矛盾：第 1 轮推荐 A，第 3 轮用户复述需求后却推荐与 A 互斥的 B | (1) 抽取 N 段多轮样本；(2) LLM-as-Judge 对每段做"一致 / 矛盾"二元判定；(3) MTC = 一致段数 ÷ N | ≥ 0.90 |
| **商品卡片字段完整率（CFC）** | 返回的商品卡片 JSON 中必填字段完整的比例。必填：name, price, image_url, recommendation_reason, citation | CFC = 字段完整的卡片数 ÷ 总卡片数 | ≥ 0.95 |
| **引用准确率（CA）** | 回答中标注的引用 `[1][2]` 是否真的指向支持该论点的文档 | (1) 抽取每条"论点-引用"对；(2) 人工/LLM 判定引用是否正确支持论点；(3) CA = 正确引用数 ÷ 总引用数 | ≥ 0.90 |

### 10.4 性能与体验指标

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|----------|--------|
| **TTFT**（首字响应时间） | 用户发送 → 第一个 token 返回 | 服务端埋点 `t_first_token - t_request_received`，统计 P50/P95/P99 | P95 ≤ 3s |
| **流式速率** | LLM 输出 tokens/s | `total_tokens / (t_last_token - t_first_token)` | ≥ 20 tokens/s |
| **端到端延迟**（E2E） | 用户发送 → 完整回复 + 卡片渲染完成 | 客户端埋点 `t_render_done - t_send` | P95 ≤ 15s |
| **卡片渲染延迟** | 收到 JSON → 完全可见 | 客户端埋点 | ≤ 500ms |
| **对话完成率**（CCR） | 用户走完完整对话路径的会话比例 | `completed_sessions / started_sessions`，"完成"定义为至少 1 次互动 + 用户主动关闭 | ≥ 70% |
| **卡片 CTR** | 商品卡片点击率 | `clicks / impressions` | ≥ 15%（参考行业基准） |

---

### 10.5 安全与合规指标

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|----------|--------|
| **拒识率**（Refusal Rate） | 被安全过滤拦截的比例 | `refused / total_queries` | 3% – 8%（过低代表过滤失效，过高代表误伤） |
| **误拒率**（False Refusal Rate） | 被拒识但实际是良性请求的比例 | 人工抽检 100 条拒识样本，标注是否误伤；`false_refusal / total_refusal` | ≤ 10% |
| **有害内容漏检率** | 应被拦截但通过的比例 | 用预制 50 条对抗样本测试，`pass_through / total_adversarial` | ≤ 5% |
| **PII 泄露率** | 回答中意外暴露用户个人信息的比例 | 自动正则 + LLM 扫描回答中的手机号 / 身份证 / 地址等 | = 0 |

### 10.6 用户反馈指标

| 指标 | 定义 | 计算方式 | 目标值 |
|------|------|----------|--------|
| **点赞率** | `likes / (likes + dislikes)` | 显式反馈聚合 | ≥ 75% |
| **点踩原因分布** | 各原因点踩占比（信息不准 / 不相关 / 太长 / 太短 / 其他） | 分类统计 | "信息不准" 占比 ≤ 30%（高则触发知识库复审） |
| **NPS**（净推荐值） | "你会推荐给朋友吗?"评分（0-10） | NPS = % 推荐者 (9-10) - % 贬损者 (0-6) | ≥ 30 |

### 10.7 评测数据集设计

**规模与构成**：

| 类别 | 数量 | 用途 |
|------|------|------|
| **核心场景标准 QA** | 200 条 | RAG 四指标 + 推荐准确率 |
| **多轮对话样本** | 50 段 | 多轮一致性 |
| **对抗 / 越狱样本** | 50 条 | 安全过滤评估 |
| **多模态样本** | 30 条 | 图片提问准确率 |
| **合计** | **330 条** | |

**单条评测样本结构**：

```json
{
  "query_id": "Q0042",
  "scenario": "商品对比",
  "category": "3C数码",
  "user_query": "对比 索尼 WH-1000XM5 和 Bose QC Ultra",
  "ideal_intent": "商品对比",
  "ideal_retrieved_doc_ids": ["DOC_001", "DOC_002", "DOC_037"],
  "ideal_relevant_skus": ["SKU_889", "SKU_902"],
  "ground_truth_answer": "两款均为旗舰降噪耳机...",
  "ground_truth_facts": [
    "WH-1000XM5 续航 30 小时",
    "QC Ultra 支持空间音频",
    "WH-1000XM5 重量 250g"
  ],
  "ideal_card_count": 2,
  "must_cite": true
}
```

**评测集构建流程**：
1. 按 4 个核心场景 × 多轮粒度采样，品类按所选数据集的实际覆盖自然分布
2. 每条由人工撰写并双人交叉审核
3. 标准答案需明确标注事实点（用于 CR / Faithfulness 计算）

---

### 10.8 评测执行流程

```
[评测触发：每次 Prompt / 检索 / 知识库变更]
              ↓
[Step 1] 跑 330 条评测集，记录每条的：
          - 检索结果（doc_ids, 顺序）
          - 生成回答 + 商品卡片 JSON
          - 各项耗时
              ↓
[Step 2] 自动计算指标：
          - RAGAS 四指标（LLM-as-Judge）
          - Precision@K / Recall@K / MRR / NDCG@K（与 ideal 对比）
          - CFC / CA（结构化检查 + LLM 判断）
              ↓
[Step 3] 生成对比报告：
          | 指标 | 上版本 | 本版本 | 变化 |
              ↓
[Step 4] 回归 Gate：
          - 任一核心指标下跌 > 5% → 阻断发布
          - 全部达标 → 通过
              ↓
[Step 5] 失败样本归档，用于 Prompt / 知识库迭代
```

### 10.9 反馈闭环闭合机制

```
用户使用产品
   ↓
反馈数据收集（显式 + 隐式）
   ↓
归类聚合（按场景 / 品类 / 问题类型）
   ↓
分流处理：
   ├── 知识库问题（信息不准 / 缺失）  → 商家补充 / 修正知识库
   ├── Prompt 问题（答非所问 / 风格差）→ Prompt 策略优化
   ├── 检索问题（漏召 / 错召）        → 检索 / 重排序参数调整
   └── 模型问题（推理 / 生成质量）    → 切换模型 / 累积 SFT 数据
   ↓
重新跑 §10.8 的离线评测
   ↓
对比新旧指标 → 通过回归 Gate → 发布新版本
```

## 12. 风险与对策

| 风险 | 严重度 | 对策 |
|------|--------|------|
| LLM API 不稳定（超时 / 限流） | 高 | 多供应商兜底 + 本地降级提示 |

| 评测数据集偏差 | 中 | 数据集覆盖 6 大场景 + 用户分类型 |
| 合规备案在比赛周期内难完成 | 中 | 比赛期间产品仅供赛事评审使用，不向公众开放，备案作为未来上线前提 |
| 流式速率不达 20 tokens/s | 低 | 选 DeepSeek-V3 等高吞吐模型，关闭非必要插件 |
