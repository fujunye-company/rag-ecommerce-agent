## 1. 核心原则

```
MVP 不是另起炉灶，而是从全量系统中截取最小闭环。
```

全量系统目标：

```
意图理解 → RAG 检索 → 商品推荐 → 商品卡片 → 多轮对话 → 多模态输入 → 反馈评测闭环
```

MVP 截取目标：

```
文字输入 → 基础意图理解 → RAG 检索 → 商品推荐 → SSE 流式回答 → 商品卡片展示 → 基础反馈
```

MVP 和全量开发必须共用以下内容：

```
1. 共用同一个 backend/ 工程
2. 共用同一个 android/ 工程
3. 共用同一套 API 路径
4. 共用同一套商品数据模型
5. 共用同一套会话数据模型
6. 共用同一套 SSE 事件格式
7. 共用同一套前端页面结构
8. 共用同一套 Agent / RAG 服务分层
```

**绝对禁止**：

```
1. MVP 单独写一个临时后端
2. MVP 单独写一套临时接口
3. MVP 用假 UI，后期全部推翻
4. MVP 商品卡片字段和全量版本不一致
5. MVP 的 RAG / Agent 逻辑后期无法复用
6. MVP 为了演示绕过真实 backend 与 android 通信链路
```

## 2. 项目工程结构

项目代码统一放在 `02-source-code/` 下，包含两个独立工程：

```
02-source-code/
├── backend/          ← 后端工程：Python FastAPI
│   ├── app/
│   │   ├── main.py          API 入口
│   │   ├── api/             路由层
│   │   ├── core/            配置 / 数据库 / 中间件
│   │   ├── schemas/         数据模型（Pydantic）
│   │   ├── models/          ORM 模型
│   │   └── services/        Agent / RAG / 商品 / 反馈 / 评测
│   ├── tests/               后端测试
│   ├── data/                商品数据 / 文档
│   ├── pyproject.toml       依赖管理
│   └── README.md
│
└── android/          ← 前端工程：Kotlin Compose
    └── app/
        ├── build.gradle.kts 依赖配置
        └── src/
            ├── main/
            │   ├── AndroidManifest.xml
            │   └── java/.../
            │       ├── ui/          UI 层
            │       ├── data/        数据层
            │       └── viewmodel/   业务逻辑
            └── test/                单元测试
```

两个工程独立编译运行：

```
backend：uvicorn 启动 FastAPI HTTP 服务（端口 8000）
android：Android Studio 编译运行 APK
通信方式：Android 通过 HTTP / SSE 接口调用后端
```

## 3. 后端工程规约

### 3.1 backend 职责

```
1. 提供 HTTP / SSE API
2. 管理商品数据、会话数据、用户反馈数据
3. 调用 Agent 工作流
4. 调用 RAG 检索服务
5. 调用大模型 OpenAPI
6. 返回结构化商品卡片数据
7. 提供评测与反馈闭环基础能力
```

### 3.2 backend 分层

```
app/
├── main.py          API 入口，挂载路由、CORS、异常处理器
├── api/             路由层 → 只处理请求/响应，不写复杂业务逻辑
├── core/            配置、数据库连接、环境变量、中间件、通用工具
├── schemas/         Pydantic 请求体/响应体定义
├── models/          ORM 数据库模型
└── services/        Agent / RAG / 商品 / 反馈 / 评测业务逻辑
```

### 3.3 后端开发约束

```
1.  api/ 中不得写 Agent 或 RAG 复杂逻辑
2.  services/ 返回值必须经过 schemas/ 校验后再传出
3.  大模型调用必须封装为独立模块（如 services/llm_client.py），禁止散落
4.  商品卡片返回结构必须稳定，不允许随意改字段名——改动需前后端同步
5.  SSE 事件类型统一在 schemas/sse_events.py 维护
6.  环境变量、API Key、数据库地址不得硬编码 → 用 .env + pydantic-settings
7.  所有核心接口必须有对应的测试文件
8.  异常必须有统一的 HTTP 异常处理器，不暴露堆栈到客户端
9.  所有外部调用（LLM API、向量库）必须设 timeout，不得无限等待
```

### 3.4 后端文件命名

```
路由文件：   api/products.py, api/chat.py, api/feedback.py
服务文件：   services/agent.py, services/rag.py, services/llm_client.py
Schema：    schemas/product.py, schemas/chat.py, schemas/feedback.py
Model：     models/product.py, models/session.py, models/feedback.py
测试文件：   tests/test_products.py, tests/test_chat.py（与源文件对应）
```

### 3.5 后端依赖管理

```
依赖声明：pyproject.toml（uv 管理）
锁定文件：uv.lock（提交到 Git）
虚拟环境：项目根目录 .venv/（不提交）
```

## 4. Android 工程规约

### 4.1 android 职责

```
1.  提供用户输入界面（文字 + 图片入口）
2.  调用后端 /api/chat 或 /api/chat/stream
3.  接收 SSE 流式返回
4.  展示 AI 流式回答
5.  渲染商品卡片
6.  展示商品详情
7.  提交用户反馈
8.  覆盖加载、错误、空状态
```

### 4.2 android 分层

```
java/.../
├── ui/          UI 层：Compose 页面、组件、Preview
├── data/        数据层：API Client（Retrofit/OkHttp）、DTO、Repository
└── viewmodel/   业务逻辑：页面状态、SSE 管理、请求调度
```

### 4.3 Android 开发约束

```
1.  UI 层不直接调用网络请求 → 通过 ViewModel
2.  ViewModel 管理页面状态，使用 StateFlow
3.  data 层封装 API 请求和 DTO 转换
4.  商品卡片字段必须与后端 schemas 对齐 → 变化时前后端同步 PR
5.  SSE 事件解析必须按 type 字段分发，预留未知 type 处理
6.  网络错误、流式中断、空结果必须有前端状态提示
7.  MVP 页面结构必须能扩展到全量版本
8.  字符串硬编码统一到 strings.xml，不得散落
```

### 4.4 Android 依赖管理

```
依赖声明：app/build.gradle.kts
Gradle Wrapper：gradle/wrapper/（提交到 Git）
编译 SDK：compileSdk 34 / targetSdk 34
最低版本：minSdk 26（Android 8.0）
```

## 5. 全量系统功能总链路

```
用户文字 / 图片输入
↓
Android Kotlin 客户端采集输入
↓
FastAPI 接收请求（POST /api/chat/stream）
↓
LangGraph Agent 工作流
↓
意图理解 → 需求槽位抽取
↓
是否需要追问？（是 → SSE clarification 事件）
↓
商品结构化过滤（品类/价格/品牌）
↓
LlamaIndex RAG 知识检索
↓
Qdrant / Chroma 向量召回 + PostgreSQL 商品查询
↓
商品推荐 / 商品对比 / 商品咨询
↓
大模型生成回答
↓
结构化商品卡片输出（product_cards 事件）
↓
SSE 流式返回 Android（text_delta → product_cards → done）
↓
Android 渲染文本、商品卡片、对比结果
↓
用户反馈（POST /api/feedback）
↓
质量评测（RAGAS 指标）
↓
Prompt 与知识库迭代
```

## 6. MVP 截取范围

### 6.1 MVP 保留链路

```
用户文字输入
→ Android 采集
→ FastAPI 接收
→ 基础 Agent 流程
→ 基础意图理解
→ 商品数据查询
→ 基础 RAG 检索
→ 商品推荐
→ 大模型生成回答
→ 商品卡片结构化输出
→ SSE 流式返回
→ Android 渲染回答与商品卡片
→ 用户基础反馈
```

### 6.2 MVP 暂缓但结构必须预留

```
1. 图片输入：MVP 可先不做，但 upload 接口路由和前端入口需预留
2. 商品对比：MVP 可先做简单版，复杂对比全量补齐
3. 多轮记忆：MVP 仅保留当前 session，全量再做长期偏好
4. 知识库版本：MVP 仅做一次入库，全量再做版本管理
5. 质量评测：MVP 仅做基础测试集，全量再做完整指标
6. 管理端：MVP 不做后端管理页面，但保留文档上传接口位置
```

### 6.3 MVP 与全量模块对照

| 模块 | MVP 截取 | 全量开发 | 共用? |
|------|---------|---------|-------|
| Android 工程 | 对话页、输入框、流式文本、商品卡片 | 首页、图片上传、详情页、对比页、反馈页、历史会话 | ✅ |
| FastAPI 工程 | health、chat、chat/stream、products、feedback | upload、compare、eval、admin、knowledge-base | ✅ |
| Agent 流程 | 基础意图识别、检索、推荐、回答 | 追问、多轮、对比、图片理解、反馈优化 | ✅ |
| RAG | 基础商品文档检索 | 文档上传、切分、向量化、引用、版本管理 | ✅ |
| 向量库 | Chroma / Qdrant 最小集合 | Qdrant 完整元数据过滤与多版本索引 | ✅ |
| 数据库 | SQLite / PostgreSQL 基础表 | 完整商品、会话、反馈、评测、知识库版本表 | ✅ |
| SSE | text_delta、product_cards、done、error | intent、retrieval、compare、clarification、feedback | ✅ |
| 商品卡片 | 名称、价格、图片、推荐理由 | 标签、评分、库存、优惠、知识来源、适配度 | ✅ |
| 评测 | 人工测试 + 少量用例 | 自动评测、指标报告、Prompt 对比、知识库迭代 | ✅ |

## 7. API 与通信规约

### 7.1 基础接口

MVP 阶段至少保留：

```
GET  /health                        → 健康检查
POST /api/chat                      → 普通问答（JSON 请求/响应）
POST /api/chat/stream               → 流式问答（SSE）
GET  /api/products                  → 商品列表（支持 ?category=&min_price=&max_price=）
GET  /api/products/{product_id}     → 商品详情
POST /api/feedback                  → 用户反馈（{session_id, product_id, rating, reason}）
```

全量阶段扩展：

```
POST /api/upload/image              → 图片上传
POST /api/products/compare          → 多商品对比
POST /api/documents/upload          → 知识文档上传
POST /api/knowledge/ingest          → 知识库入库
POST /api/eval/run                  → 触发评测
GET  /api/eval/report               → 评测报告
```

### 7.2 API 通用规范

```
1. 所有 HTTP JSON 响应体统一包裹：{ "code": 0, "data": {...}, "message": "ok" }
2. SSE 流式事件不适用此包裹格式，遵循 §7.3 逐事件 JSON 格式
3. 错误响应：{ "code": 4xxx/5xxx, "data": null, "message": "具体错误描述" }
4. 列表接口支持分页：?page=1&size=20
5. 所有接口（除 /health）需记录请求日志
6. 生产环境关闭 FastAPI 自动生成的 /docs（或加基础认证）
```

### 7.3 SSE 事件类型

MVP 必须支持：

```
text_delta      → { "type": "text_delta", "content": "..." }
product_cards   → { "type": "product_cards", "products": [...] }
done            → { "type": "done" }
error           → { "type": "error", "message": "...", "code": "..." }
```

全量扩展：

```
intent          → 意图识别结果
retrieval       → 检索过程透明化
compare         → 对比结果
clarification   → 需要追问
feedback        → 反馈确认
```

### 7.4 SSE 事件示例

```json
{"type": "text_delta", "content": "根据你的预算和宿舍使用场景，"}
```

```json
{
  "type": "product_cards",
  "products": [
    {
      "id": "p001",
      "name": "便携蓝牙音箱 A",
      "price": 199,
      "image_url": "https://example.com/p001.jpg",
      "reason": "价格在 300 元以内，体积较小，适合宿舍使用"
    }
  ]
}
```

```json
{"type": "done"}
```

```json
{"type": "error", "message": "检索服务暂时不可用，请稍后重试", "code": "RAG_UNAVAILABLE"}
```

## 8. 商品卡片数据规约

MVP 最小字段：

```json
{
  "id": "p001",
  "name": "便携蓝牙音箱 A",
  "price": 199,
  "image_url": "https://example.com/p001.jpg",
  "reason": "适合宿舍场景，价格符合预算"
}
```

全量扩展字段：

```json
{
  "id": "p001",
  "name": "便携蓝牙音箱 A",
  "brand": "品牌 A",
  "category": "数码 / 音箱",
  "price": 199,
  "stock": 20,
  "rating": 4.7,
  "sales": 1200,
  "image_url": "https://example.com/p001.jpg",
  "tags": ["宿舍", "便携", "低音"],
  "reason": "适合宿舍场景，价格符合预算",
  "evidence": ["商品详情-低音表现", "用户评价摘要-便携性"],
  "match_score": 0.87
}
```

规约：

```
1. MVP 可以只使用最小字段
2. 全量字段必须向后兼容 MVP 字段（新增字段设默认值）
3. Android ProductCard data class 应优先兼容全量字段
4. 后端严禁随意更改 id / name / price / image_url / reason 字段名
5. 字段变更需前后端同步发 PR，并在 CHANGELOG 记录
```

## 11. Git 工作流

### 11.1 分支策略

```
main            ← 受保护分支，禁止直接推送。仅通过 GitHub PR squash merge 合入
feature/xxx     ← 功能分支，所有开发在此进行（如 feature/add-chat-stream）
fix/xxx         ← 修复分支（如 fix/sse-reconnect）

每次修改流程：
1. git checkout -b feature/xxx
2. git add + git commit
3. git push origin feature/xxx
4. GitHub Web 创建 PR → review → squash merge → main
```

### 11.2 Commit 格式

```
<type>(<scope>): <subject>

类型：feat / fix / docs / refactor / test / chore
范围：backend / android / docs / data
主题：中文，≤50 字

示例：
feat(backend): 实现 /api/chat/stream SSE 流式接口
fix(android): 修复商品卡片图片加载失败导致崩溃
docs(backend): 更新 API 文档，补充 SSE 事件说明
test(backend): 添加 RAG 检索服务的单元测试
```

### 11.3 合并要求

```
1. feature 分支合并到 develop 前必须通过 CI（如有）
2. develop 合并到 main 前必须 code review
3. 禁止 force push 到 main / develop
4. PR 标题遵循 commit 格式
5. PR 描述写清楚：做了什么 / 怎么测试 / 影响范围
```

## 13. 测试要求

### 13.1 测试金字塔

```
        ╱  E2E  ╲        全链路测试（可选，MVP可不做）
       ╱──────────╲

    ╱    单元测试     ╲    核心服务函数 / 数据模型校验

### 13.2 后端测试

```

2. 每个 API 路由至少有 1 个集成测试
3. RAG 检索必须有准确性测试（≥5 条测试用例）

```
1. ViewModel 状态逻辑必须有单元测试
2. DTO 反序列化必须有测试（覆盖所有字段）
3. SSE 事件解析必须有测试（覆盖所有 type）
```

## 14. 错误处理

### 14.1 后端错误处理

```
1. 所有异常通过 FastAPI exception_handler 统一捕获
2. HTTP 异常返回标准格式：{ "code": xxx, "message": "...", "data": null }
3. 错误码体系：
   - 4xx：客户端错误（参数校验、权限）
   - 5xx：服务端错误（LLM 调用失败、数据库异常）
4. 生产环境不暴露堆栈信息，返回通用错误描述
5. 关键异常记录日志并附带 request_id 用于追踪
```

### 14.2 Android 错误处理

```
1. 网络请求必须有 timeout 和重试策略
2. SSE 断连必须有自动重连（指数退避，最多 3 次）
3. 商品图片加载失败显示占位图
4. 后端返回 error 事件时，UI 展示 friendly 提示而非原始错误
```

## 15. 日志规范

```
后端日志级别：
  DEBUG   → 开发调试信息
  INFO    → 请求日志（method、path、status、duration）
  WARNING → 可恢复异常（LLM 调用重试、向量库超时）
  ERROR   → 不可恢复异常（数据库宕机、API Key 无效）

日志格式：
  [时间] [级别] [request_id] 消息

必须记录的场景：
  1. 每个 HTTP 请求（INFO）
  2. LLM API 调用（INFO，含耗时和 token 数）
  3. RAG 检索（DEBUG，含召回数量和分数）
  4. SSE 连接建立/断开（INFO）
  5. 异常（ERROR，含 request_id 和简要原因）
```

## 16. 安全规范

```
1.  API Key / 数据库密码 / Secret 不得提交到 Git → 用 .env + .gitignore
2.  .env.example 提供模板，不含真实值
3.  用户上传的图片需要校验类型和大小（≤10MB，仅允许 jpg/png/webp）
4.  LLM 生成内容需做敏感词过滤（政治、色情、暴力）
5.  商品推荐不得因性别/地域/年龄产生歧视性差异
6.  SQL 查询使用参数化，禁止拼接用户输入
7.  CORS 配置仅允许 Android 客户端来源
8.  生产环境关闭 /docs 或加基础认证
9.  会话数据需遵守 PIPL，提供用户删除入口
10. 如面向公众开放，必须完成生成式 AI 服务安全评估和算法备案
    （注：比赛期间如仅内部演示/答辩使用，不构成"面向公众"，备案非强制；
    但代码和架构需预留合规接口，以便未来产品化时快速对接）
```

## 17. 性能要求

```
1. /health 响应时间 < 100ms（P99）
2. /api/chat 首字延迟 < 3s（SSE 第一个 text_delta）
3. SSE 流平均推送速率 ≥ 20 tokens/s
4. RAG 检索延迟 < 2s（向量检索 + 重排序）
5. 商品卡片渲染不超过 500ms（从收到事件到 UI 展示）
6. Android APK 体积 ≤ 50MB
```
